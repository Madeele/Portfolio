Salvatio Push app Prototype

Prototype created in collaboration with Wojciech Dzwonczyk, while having an Internship in Salvatio Push company. As a goal, we set creating a solution that will be a perfect support for Salvatio Push device, the main company's product.

Role: UI Designer, UX Designer

Program: Adobe XD

In Team with: Wojciech Dzwończyk

Problem statement:

Salvatio Push start-up created a device that saves people lifes by sending messages with help request in case of danger. Just with one button click every person can be sure, that someone will come and give help if needed.

The problem was, that to have the device fully working, Salvatio Push also needed an app that will be the easiest thing to use by all the customers. It has a basic design but it wasn't adjusted for the target group and it was missing a lot of funcionalities:

- Notifications in the app
- Settings
- Profile page
- Map with potential needy people

There was also a huge lack of information of how the app works and what are the rules of payments, which is a basic for the client to understand, before using the App.

2Gather Web App

Web App created as a group school Exam Project for the company that we come up with. It's a solution just for mobile phones where we used Firebase and Mail/Facebook authentication.

Role: UI Designer, UX Designer, Front-end Developer

Program: Illustrator, Adobe XD, Visual Studio Code, Github, GitKraken

In Team with: Dan Berger, Aleksandra Wytulany and Kristine Dzumakajeva

Problem statement:

While making a Concept test with our fellow students, we tried to get the knowledge about the Events Apps in Aarhus (If there are any? What they offer? What is missing?) We asked around 10 people about their opinion on this topic. It came out that Facebook is their only option and they really hate it. It's very difficult to understand Facebook events' flow and find specific event that interest our user.

We asked them what is the most important for them in finding an event and they answered:

1.Distance
2.Date
3.How many friends are attending

The other problem was how difficult it is to find releavant information about the event and find the exact category that interest you.

PROCESS

Before creating every project final solution, I'm making a full desk and field research. As I focus on design and web development fields in my portfolio, I'm not attaching this parts of the process down below.

To start the design part, we firstly have to set up a base and the whole foundation, so every member of the group will know which colors we are going to use or how our buttons will look like. We already wanted to also agree on the fonts for headings and normal text for our solution, not to spend a lot of time on searching for it later.Everyone of us had a task to search for some inspirations that we want to use while creating our own project, so we gathered some mobile solutions to make an agreement on one style.

For creating those foundations, we prepared a graphical visualization of what we have in our mind in the Style-tile.

Thanks to wireframes, we got a layout for all the elements that we wanted to implement and as we didn’t have a lot of sketches, these wireframes were really important for our UX quality. We spend a lot of time discussing where to put some boxes or how to construct pages that we need.

Then there was a perfect time to use our style-tiles and create mockups that we will also use later as the prototype. We made an AB testing to find out if we should use pictures in our ‘Events’ or only a color. Images came out to be more interesting for our users so our decision was even more supported as we would have to many colors on pages. We used our base colors for creating decorations and Logo for 2Gather App as well as buttons and categories boxes.

CODING

The main focus in this project was placed on the Single Page Application layout of our website and usage of JavaScript. We had few main functionalities in our Web App that I would like to highlight:
• Events ordered by 3 properties (date, distance and friends attending).
• Search bar that gives content filtered by name of the event.
• Categories that are dividing the events e.g. ‘music’, ‘party’.
• Log-in through Facebook and E-mail, that enables welcoming a user with his name and displaying his e-mail in profile page.

Let me start right away with describing some examples of the code that I was responsible for.

To append our events, I used the filter() method placed inside of the onSnapshot() method, so we would get all the updates from Firestore in real time. I created a new array that will hold filtered events based on the value that user will put into search bar input. Then I appended events that consist of the written value by appendCategories() function which we created and used before. I had to play more with the code as I wanted to show ‘Categories’ again if the array won’t contain searched value, so I created an ‘if else’ function that checks if the value consists of searched elements, and If doesn’t – it runs a function that shows the categories and search bar again and hides a section with filtered events.

As I was also responsible for user authorization through Facebook and Email, I spend a lot of time on reading docs and getting the knowledge about appending current user data to our code. I managed to get current user name and email, as well as profile FB picture in some cases. Maybe it doesn’t look difficult on the screenshot, but it took tones of time and focus to understand the “permissions” on Facebook developers page and the work of id’s of every user. After hours of understanding the whole process, it came out that the code for it is just so simple as you can see above.

Travelday website

Prototype of the App and website created for Travelday company.

Role: UI Designer, UX Designer, Front-end Developer

Program: Illustrator, Adobe XD, Visual Studio Code, Github, GitKraken

In Team with: Wojciech Dzwończyk, Aleksandra Wytulany and Raluca Calin

Problem statement:

After talking with our Client - Travelday, we knew that our users weren't held on the website due to lack of personal connection to it. They haven’t felt that they need to come back to the website that doesn't create personal values for them. There is no option of creating the profile, nor saving your searches or adding to favorites anything that you found interesting.

World has no barriers for us right now so travel providers and agencies has to stand out to be known and recognizable, as there is a wide range of all kinds of travel companies that wants to get our attention. That’s how the first question appeared in our minds :

• What’s unique about Travelday and their solution, how are they different from their competitors?

If there is so many travel providers, how exactly we can achieve this goal knowing that we are searching among two target groups that Ralu will tell you more about in further chapters. As we are aiming for Generation Z and Millennials, we have to somehow make them more aware of websites that give a true value for their users. But how to make Travelday one of them? Here the next question was born :

• How can we best create awareness and interest among the target groups of Generation Z and Millennials – and what are they interested in?

Thinking about awareness we could automatically head toward traffic for Travelday website, solution or maybe even social media? All the triggers for coming back to the websites are created by need of achievements or human addictions/behavior so in our solution we also have to think:

• How can we increase the traffic for Travelday's solution/website/social media and make users stay there and even come back later?

As the last thing after all our questions and brainstorming we also had to think about how to implement and include all those findings into design of our solution. As we mainly created the problem statement, we can now think what is limiting us in our solution.
